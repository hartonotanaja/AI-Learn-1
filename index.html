How Large Language Models Actually Work

Before you write a prompt or call an API, you need to understand what you are working with. An LLM is not a search engine, not a database, and not a human. It is a very specific kind of mathematical machine, and knowing how it works will make you dramatically better at using it.

Core Idea

An LLM is trained to predict the most likely next token (word piece) given everything that came before it. Everything — reasoning, creativity, code generation — emerges from this one simple task done at massive scale.

1.1 — Tokens: The Atomic Unit

LLMs don't read characters or words — they read tokens. A token is roughly 3–4 characters on average. The word "learning" might be one token; "unbelievable" might be three ("un", "believ", "able").

Why does this matter to you? Because:

  • API costs are measured in tokens (input + output)
  • Context windows are measured in tokens (e.g., "200K context" = 200,000 tokens)
  • Unusual words, code, and non-English text cost more tokens per character
  • Numbers like "1,234,567" tokenize differently than you expect
Example Token breakdown 
# The text "Hello, world!" breaks into tokens like:
"Hello"  → 1 token
","      → 1 token
" world" → 1 token
"!"      → 1 token
# Total: 4 tokens

# But "antidisestablishmentarianism" might be:
"ant" "idis" "establish" "ment" "arian" "ism"
# Total: 6 tokens

1.2 — The Transformer Architecture (Simplified)

Modern LLMs are based on the Transformer architecture (introduced in the 2017 paper "Attention Is All You Need"). You don't need to implement one, but understanding the key ideas helps.

🔢

Embeddings

Each token is converted into a list of hundreds of numbers (a vector). Similar concepts end up with similar vectors — "king" and "queen" are numerically close.

👁️

Self-Attention

The mechanism that lets the model relate any token to any other token in the context. This is how it understands "it" refers to "the cat" three sentences back.

🏗️

Layers

A transformer stacks many attention layers. Early layers detect grammar; deep layers understand semantics, logic, and concepts. GPT-4 has ~100+ layers.

🎲

Temperature

Controls how "random" the next-token selection is. Temperature=0 is deterministic (always picks the top token). Temperature=1 introduces creativity and variation.

1.3 — Training: How a Model Learns

Training an LLM happens in stages. Understanding each stage helps you know why a model behaves the way it does.

1

Pre-training (Self-supervised)

The model reads hundreds of billions of tokens from the internet, books, and code. It learns to predict the next token, developing a broad world model in the process. This is by far the most expensive stage — it costs tens of millions of dollars and weeks of GPU time.

2

Supervised Fine-tuning (SFT)

The pre-trained model is fine-tuned on high-quality human-written examples of good conversations. This teaches it to respond helpfully in the format of a chat assistant, not just predict random internet text.

3

RLHF / Constitutional AI

Human raters (or the model itself via a "Constitution") rank different outputs. The model learns from this feedback to be more helpful, harmless, and honest. This is what makes Claude feel different from a raw pre-trained model.

Why This Matters for You

A pre-trained model doesn't "want" to help you — it just predicts tokens. Fine-tuning and RLHF are what turn that raw capability into an assistant that follows instructions, refuses harmful requests, and maintains a coherent persona. When you're confused why a model behaves strangely, thinking about this pipeline often explains it.

1.4 — The Context Window: Short-Term Memory

An LLM has no persistent memory. Every time you call it, it reads everything in the context window from scratch. The context window is the total amount of text it can "see" at once — your system prompt, the conversation history, documents you've pasted, and its own response so far.

Model Context Window Approx. Pages of Text
GPT-4o 128K tokens ~96 pages
Claude Sonnet 4.6 200K tokens ~150 pages
Gemini 1.5 Pro 1M tokens ~750 pages
DeepSeek R2 128K tokens ~96 pages

Important nuances: even with a 200K context window, models tend to pay more attention to text at the beginning and end of the context. Content buried in the middle can get "lost". This is called the lost in the middle problem — a key engineering consideration when building RAG systems.

1.5 — What Models Can and Cannot Do

LLMs Are Good At LLMs Struggle With
Summarizing and rewriting text Precise arithmetic (they approximate)
Generating structured output (JSON, XML) Real-time or factual current information
Code generation and debugging Counting exact characters in long strings
Reasoning step-by-step when prompted Knowing they don't know something
Following complex instructions Staying consistent over very long outputs
Role-playing and persona maintenance Precise spatial or logical puzzles

Prompt Engineering: The Core Skill

Prompt engineering is the practice of designing inputs to language models to reliably get the outputs you want. It is part science, part craft. Good prompting can be the difference between a useful tool and a frustrating one — even with the exact same model.

Mental Model

Think of a prompt as instructions to a very smart contractor. If you vague out ("make it good"), you'll get vague results. If you specify the goal, the format, the constraints, and an example of what "good" looks like, you'll get something useful.

2.1 — Anatomy of a Prompt

A well-structured prompt usually has several components. Not every prompt needs all of them, but knowing each one gives you tools to pull from.

Prompt Structure Full anatomy 
# 1. ROLE / PERSONA (optional but powerful)
You are a senior data analyst at a SaaS company.

# 2. CONTEXT (what the model needs to know)
The following is a CSV export of monthly churn data 
from our customer database for Q1–Q3 2025.

# 3. TASK (what you want done — be specific)
Analyze this data and identify the top 3 trends
driving customer churn.

# 4. FORMAT (how you want the output structured)
Respond with:
- A one-paragraph executive summary
- 3 bullet points, each with: trend name, evidence, 
  and a recommended action

# 5. CONSTRAINTS (what to avoid)
Do not speculate beyond what the data shows.
Keep the summary under 100 words.

# 6. EXAMPLE (optional "few-shot" — shown in Module 3)

2.2 — System Prompts vs. User Prompts

When using the API, you interact with two kinds of messages:

⚙️

System Prompt

Sets persistent instructions for the entire conversation. Think of it as the model's "job description." It persists across turns. Ideal for: persona, format rules, domain context, constraints.

💬

User Prompt

The actual message sent per turn. Can override or extend system instructions. Ideal for: the specific task, data to process, or question to answer.

Python / API System + User prompt structure 
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system="""You are a concise technical writer.
Always respond in plain English.
Format all lists as numbered lists.
Never use jargon without explaining it.""",
    messages=[
        {
            "role": "user",
            "content": "Explain what an API is to a non-technical manager."
        }
    ]
)

print(response.content[0].text)

2.3 — The Seven Principles of Good Prompting

1

Be specific about what you want

Vague: "Write something about climate change." → Specific: "Write a 200-word explainer of carbon sequestration for a high school student, using one analogy."

2

Specify the output format

Tell the model exactly what shape the output should take: JSON, markdown, a numbered list, a table, a paragraph, a specific number of items. Models will guess otherwise, often incorrectly.

3

Give it a role or persona

"You are a senior Python developer reviewing code for security vulnerabilities" produces more focused output than no role. The persona primes the model's knowledge.

4

Separate instructions from data

Use XML tags or clear delimiters to separate what you want done from the data you're providing. This dramatically reduces prompt injection risk and improves parsing.

5

Tell it what NOT to do

Negative constraints are powerful. "Do not make up statistics" or "Do not use bullet points" often work better than trying to describe every aspect of what you want positively.

6

Ask it to think before answering

For reasoning tasks, add "Think through this step by step before giving your final answer." This is the basis of Chain-of-Thought prompting (Module 3).

7

Iterate — prompting is experimental

Your first prompt is a hypothesis. Run it, see what breaks, adjust. Build a personal library of prompt patterns that work for your use cases.

2.4 — Using XML Tags (Claude-Specific Superpower)

Claude is specifically trained to understand XML-style tags in prompts. Using them to separate sections makes your prompts dramatically more reliable.

Prompt XML structured prompt 
You are a document analyst. Review the document below and answer the question.

<document>
Quarterly revenue grew 12% YoY to $4.2M. Gross margin improved 
from 61% to 67% driven by infrastructure optimization. 
Customer count reached 1,240, up from 890.
</document>

<question>
What drove the gross margin improvement?
</question>

<format>
Answer in one sentence. Cite the specific figure from the document.
</format>
Pro Tip

The tags don't need to be official XML — you invent them for your use case. Common useful tags: <context>, <task>, <format>, <example>, <document>, <constraints>, <thinking>. Claude treats content inside different tags as distinct semantic sections, not just text.

Prompting Techniques: Zero-Shot, Few-Shot & Chain-of-Thought

These are the three foundational prompting strategies used in almost every serious LLM application. You'll combine them constantly in practice.

Basic
Zero-Shot Prompting

You give the model a task with no examples. You rely entirely on knowledge baked in during training. Works well for common, well-defined tasks.

ExampleZero-shot
Classify the sentiment of the following review as 
POSITIVE, NEGATIVE, or NEUTRAL.

Review: "The interface is clean but the app crashes every morning."

Sentiment:

When to use it: Routine tasks, well-known formats, when you don't have examples ready. Start here — add examples only if results are inconsistent.

Intermediate
Few-Shot Prompting

You provide 2–5 examples of input → output pairs before giving the actual input. This "shows" the model what you want, dramatically improving consistency for tasks that are unusual or format-specific.

ExampleFew-shot with 2 examples
Extract the company name and funding amount from each sentence.
Return as JSON.

Input: "Acme Corp raised $5M in Series A funding last week."
Output: {"company": "Acme Corp", "amount": "$5M", "round": "Series A"}

Input: "TechFlow announced a $20 million seed round led by Sequoia."
Output: {"company": "TechFlow", "amount": "$20M", "round": "Seed"}

Input: "DataNova closed its $8M Series B yesterday with 12 investors."
Output:

Key rules for good few-shot examples:

  • Cover edge cases, not just easy examples
  • Keep format identical across all examples
  • 3–5 examples is usually the sweet spot; more rarely helps
  • Order matters: put the most representative example last

When to use it: Custom output formats, domain-specific tasks, when zero-shot gives inconsistent results, classification with unusual categories.

Advanced
Chain-of-Thought (CoT) Prompting

You instruct the model to reason step by step before giving a final answer. This unlocks dramatically better performance on multi-step reasoning, math, logic, and complex decisions. The key insight: the model is allowed to "think out loud" in its output, which improves the final answer.

ExampleWithout CoT (often wrong)
Q: A store has 120 apples. They sell 45 on Monday, 
receive 30 more on Tuesday, and sell half their 
remaining stock on Wednesday. How many apples 
do they have at the end?

A:
# Model often guesses or makes arithmetic errors
ExampleWith CoT (reliable)
Q: A store has 120 apples. They sell 45 on Monday, 
receive 30 more on Tuesday, and sell half their 
remaining stock on Wednesday. How many apples 
do they have at the end?

Think through this step by step, then give your final answer.

A:

# Model output:
# Step 1: Start with 120 apples.
# Step 2: Sell 45 on Monday → 120 - 45 = 75 apples.
# Step 3: Receive 30 on Tuesday → 75 + 30 = 105 apples.
# Step 4: Sell half on Wednesday → 105 / 2 = 52.5
#         (round down to 52, as apples are whole)
# Final Answer: 52 apples.

Variants of Chain-of-Thought:

  • Zero-shot CoT: Just add "Let's think step by step." — simple but effective.
  • Few-shot CoT: Provide examples that include the step-by-step reasoning in the output. More powerful but requires more prompt space.
  • Extended Thinking (Claude): Claude has a built-in "thinking" mode where it reasons internally before responding — deeper than standard CoT and doesn't consume your output tokens.

When to use it: Math, multi-step logic, decision trees, debugging, anything where you need the model to not "jump to an answer".

3.4 — Combining Techniques

In practice, you'll combine these. Here's a real-world example of all three together:

Real-world Prompt Zero-shot + CoT + Format control 
You are a senior product manager reviewing user feedback.

<task>
Classify each piece of feedback into one of these categories:
BUG_REPORT, FEATURE_REQUEST, COMPLIMENT, QUESTION

Think through each classification briefly, then give your answer.
Return a JSON array.
</task>

<feedback_items>
1. "The export to CSV button doesn't work on Firefox."
2. "Would love a dark mode option."
3. "Best project management tool I've used in years!"
4. "How do I invite team members to a workspace?"
</feedback_items>

<output_format>
[
  {"id": 1, "reasoning": "...", "category": "..."},
  ...
]
</output_format>
Pro Tip — Prompt Caching

When using Claude's API, long system prompts can be cached. If you're processing many items with the same instructions, caching the system prompt saves ~90% of input token costs. This becomes important as your prompts grow in size and your usage scales up.

✦ Exercise 3.1 — Build a Chain-of-Thought Prompt

Design a CoT prompt for a real problem you face

Think of a real decision or analysis task in your work. Design a Chain-of-Thought prompt for it.

  1. Define the task (e.g., "Decide whether a feature request should be prioritized")
  2. List the criteria the model should reason through (e.g., user impact, effort, strategic fit)
  3. Ask it to reason step by step through each criterion before giving a final recommendation
  4. Specify the output format (e.g., JSON with keys: reasoning, score, recommendation)
  5. Test it with two or three real examples and observe how the reasoning changes the quality of the answer

What Are AI Agents — and How Do They Think?

Up until now we've been thinking about a single prompt → single response. An agent is when an LLM is given tools and can take actions in a loop until a task is complete. This is the bridge from Phase 1 to the rest of the learning path.

Definition

An AI agent is an LLM that has been given: (1) a goal to achieve, (2) tools it can use to take actions, and (3) the ability to observe the results of those actions and decide what to do next — in a loop.

4.1 — The Agent Loop

Every agent — regardless of the framework (LangChain, CrewAI, Claude native) — follows the same fundamental loop:

1

Perceive

The agent receives input: a task, conversation history, tool results, documents, or any other context stuffed into its context window.

2

Think (Reason)

Using CoT-style reasoning, the agent decides: What is my goal? What do I know? What do I still need? Which tool should I use next?

3

Act (Tool Call)

The agent outputs a structured "tool call" — e.g., search the web, write a file, call an API, run code. The framework executes this and returns the result.

4

Observe

The tool result is added back to the context. The agent now has new information to reason with.

5

Repeat or Finish

The agent decides: Is my task complete? If yes, output the final answer. If no, go back to step 2 and repeat with the new information.

Pseudocode The agent loop 
while task_not_complete:
    thought = llm.reason(context, available_tools)
    
    if thought.action == "finish":
        return thought.final_answer
    
    tool_result = tools[thought.tool_name].run(thought.tool_input)
    
    context.append({
        "thought": thought,
        "tool_result": tool_result
    })

4.2 — The Three Core Agent Components

🧠

Planning

Breaking a complex goal into sub-tasks. Some agents do this explicitly upfront ("task decomposition"); others plan dynamically step-by-step. Chain-of-Thought prompting is the underlying mechanism.

🗄️

Memory

Short-term: the context window — everything in the current conversation. Long-term: external databases, vector stores, files the agent can write and read. Memory is what gives agents continuity across sessions.

🔧

Tool Use

Functions the agent can call: search the web, execute Python, read/write files, call APIs, send emails, query databases. Tools are what give agents hands — the ability to act in the world beyond just generating text.

4.3 — LLM App vs. Single Agent vs. Multi-Agent

Type Description Example Complexity
LLM App Fixed prompt → response. No loop, no tools. Summarize this document Low
Single Agent One LLM in a loop with tools. Can handle multi-step tasks. Research a topic and write a report Medium
Multi-Agent Multiple LLMs, each specialized. One "orchestrator" delegates to others. Researcher + Writer + Reviewer agents High
Anthropic's Advice

Start with the simplest architecture that could work. Many tasks that seem to need a multi-agent system can be solved with a well-designed single agent or even a clever prompt chain. Add complexity only when you've proven simpler approaches are insufficient.

4.4 — Your First Agent: Tool Use with Claude

Here's the minimal code to give Claude a tool and let it decide when to use it:

Python Simple tool use example 
import anthropic
import json

client = anthropic.Anthropic()

# 1. Define a tool
tools = [
    {
        "name": "get_weather",
        "description": "Get the current weather for a city.",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "The city name"
                }
            },
            "required": ["city"]
        }
    }
]

# 2. Ask Claude something that requires the tool
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    messages=[{
        "role": "user",
        "content": "What's the weather like in Jakarta right now?"
    }]
)

# 3. Check if Claude wants to use a tool
if response.stop_reason == "tool_use":
    tool_call = response.content[1]
    print(f"Claude wants to call: {tool_call.name}")
    print(f"With input: {tool_call.input}")
    # → Your code executes the real tool here
    # → Then you send the result back to Claude
    # → Claude gives the final answer to the user
